API Usage

API 使用

ccLoad 暴露标准 Anthropic、OpenAI、Gemini 与 Codex 兼容端点，客户端只需要换 base URL 和访问令牌。

Anthropic Messages OpenAI Chat Gemini v1beta Codex

调用前准备

先在管理后台创建 API 访问令牌，再把客户端 base URL 指向 ccLoad。不要把上游 Key 发给客户端，客户端只持有 ccLoad token。

登录管理后台

访问 http://localhost:8080/web/，使用 CCLOAD_PASS 登录。

添加渠道

进入渠道页，配置上游类型、URL、API Key、优先级、模型和限额。多 URL 用逗号分隔。

创建访问令牌

进入令牌页创建客户端 token，并按需限制模型、渠道或成本。没有 token 时代理接口返回 401。

端点速查

协议转换开启后，同一个上游渠道可以服务不同客户端协议。默认 upstream 模式会尽量保持上游原生语义。

客户端 端点 用途
Anthropic / Claude Code POST /v1/messages Messages API，支持流式与非流式。
OpenAI 兼容客户端 POST /v1/chat/completions Chat Completions，适合 OpenAI SDK 直接替换 base URL。
Gemini 客户端 POST /v1beta/models/{model}:generateContent Gemini generateContent 兼容端点。
Token 统计 POST /v1/messages/count_tokens 本地 Token 估算，无需转发到上游。
健康检查 GET /health 进程存活检查，适合容器健康探针。

API 调用示例

下面的 `Authorization` 使用 ccLoad 访问令牌，不是上游供应商 Key。

bash 
curl -X POST http://localhost:8080/v1/messages \
  -H "Authorization: Bearer your_ccload_token" \
  -H "Content-Type: application/json" \
  -H "anthropic-version: 2023-06-01" \
  -d '{
    "model": "claude-sonnet-4-6",
    "max_tokens": 1024,
    "messages": [
      {"role": "user", "content": "Hello from ccLoad"}
    ]
  }'
bash 
curl -X POST http://localhost:8080/v1/chat/completions \
  -H "Authorization: Bearer your_ccload_token" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-5",
    "messages": [
      {"role": "user", "content": "Hello from OpenAI-compatible client"}
    ],
    "stream": true
  }'
bash 
curl -X POST "http://localhost:8080/v1beta/models/gemini-2.5-pro:generateContent" \
  -H "Authorization: Bearer your_ccload_token" \
  -H "Content-Type: application/json" \
  -d '{
    "contents": [
      {
        "role": "user",
        "parts": [{"text": "Hello from Gemini-compatible client"}]
      }
    ]
  }'
bash 
curl -X POST http://localhost:8080/v1/messages/count_tokens \
  -H "Authorization: Bearer your_ccload_token" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "claude-sonnet-4-6",
    "messages": [
      {"role": "user", "content": "Count this locally"}
    ]
  }'

管理后台怎么用

后台不是装饰页面，它是运行时控制面：渠道配置、令牌限额、日志追踪、统计趋势和调试日志都在这里闭环。

  • 渠道页：增删改查上游渠道，导入导出 CSV，测试渠道可用性。
  • 令牌页：给不同客户端分配 token、模型范围、渠道范围和成本限制。
  • 日志页：按状态码、渠道、模型、令牌筛选请求，定位失败原因。
  • 统计页：看请求数、Token、成本、首字延迟和健康度。
  • 趋势页：按时间范围观察 RPM、成本和延迟走势。

渠道管理 API

Web 界面适合日常操作；批量导入、自动化初始化和迁移可以直接走管理 API。

添加渠道

bash 
curl -X POST http://localhost:8080/admin/channels \
  -H "Authorization: Bearer admin_session_or_token" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Claude-API",
    "type": "anthropic",
    "url": "https://api.anthropic.com",
    "api_key": "sk-ant-xxx",
    "priority": 10,
    "models": ["claude-sonnet-4-6"],
    "enabled": true
  }'

CSV 导入导出

bash 
curl -H "Authorization: Bearer your_token" \
  http://localhost:8080/admin/channels/export > channels.csv

curl -X POST -H "Authorization: Bearer your_token" \
  -F "file=@channels.csv" \
  http://localhost:8080/admin/channels/import

排障看日志，不靠猜

ccLoad 会把 499、596、597、598、599 等自定义状态码落进日志，区分客户端取消、配额超限、SSE 错误、首字超时和流中断。

429 不计入健康度成功率。它会作为限流维度单独显示，避免把“上游限流”误判成渠道坏了。
