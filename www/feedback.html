Support

Use the right channel for bugs, feature requests, usage discussions, and security reports.

Choose by issue type

The more specific the support channel, the faster the fix. Reports without logs, version information or reproduction steps usually require follow-up.

🐞

Bug Reports

Use Issues for request failures, protocol transform bugs, incorrect statistics or UI behavior problems.

  • Include the ccLoad version and deployment method.
  • Include the request endpoint, status code and relevant log snippet.
  • Describe expected behavior and actual behavior.
Submit Issue
💡

Feature Requests

For new protocols, providers, metrics, deployment support or admin console improvements, start with the use case.

  • Explain the workaround you use today.
  • Explain who the new capability serves.
  • Avoid speculative requirements that may never be used.
View Issues
💬

Usage Discussions

Deployment choices, channel configuration, model selection and cost strategy are often discussions rather than bugs.

  • Describe the deployment environment.
  • Explain the current configuration shape.
  • Share only the minimum necessary logs; never paste full secrets.
Open Discussions
🔐

Security Reports

Do not publicly post exploit details for auth bypass, sensitive data leaks, request-rule injection or TLS risks.

  • Contact maintainers through GitHub's private security channel first.
  • Provide impact scope and a minimal reproduction.
  • Do not paste upstream keys, tokens or passwords into public issues.
Security Page
🛠️

Contributing Code

Keep bug fixes and feature work focused. Open an issue first for broad refactors.

  • Go builds and tests must use -tags sonic.
  • Run make verify-web for frontend verification.
  • Do not mix unrelated formatting or generated files into the change.
