反馈支持

Bug、功能建议、使用讨论和安全问题分别走清晰渠道，别把问题埋在聊天记录里。

按问题类型选择入口

反馈入口越具体，修复越快。缺日志、缺版本、缺复现步骤的问题只能来回追问。

🐞

Bug 报告

请求失败、协议转换异常、统计口径错误、界面行为异常，都走 Issues。

  • 带上 ccLoad 版本和部署方式。
  • 贴请求端点、状态码和日志片段。
  • 说明期望行为和实际行为。
提交 Issue
💡

功能建议

新协议、新供应商、统计维度、部署支持或管理后台改进，先讲清楚使用场景。

  • 说明当前绕路方案。
  • 说明新增能力要服务谁。
  • 避免“以后可能需要”的空需求。

使用讨论

部署选型、渠道配置、模型选择、成本策略这类不一定是 Bug 的问题，走 Discussions。

  • 描述部署环境。
  • 说明当前配置结构。
  • 贴最小必要日志，不要贴完整密钥。
进入讨论区
🔐

安全问题

鉴权绕过、敏感信息泄露、请求规则注入、TLS 风险等问题不要公开贴利用细节。

  • 先在 GitHub 私密渠道联系维护者。
  • 给出影响范围和最小复现。
  • 不要把上游 Key、令牌或密码贴到公开 Issue。

贡献代码

修复 Bug 或补功能请保持改动聚焦。大范围重构先开 Issue 对齐边界。

  • Go 构建和测试必须带 -tags sonic。
  • 前端验证优先运行 make verify-web。
  • 不要混入无关格式化和生成文件。
