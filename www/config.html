Startup Environment Variables

Startup variables answer one question: can the service boot safely? Runtime policies that can be hot-updated should stay in the admin console, not in environment variables.

Variable Default Purpose Recommendation
CCLOAD_PASS None Admin console password. The service exits when it is not set. Required in production. Use platform Secrets or Docker Secrets.
CCLOAD_API_TOKENS None Pre-provisions API access tokens at startup. Supports token|description. Useful for automated recovery. Prefer the token page for normal management.
API_TOKENS None Compatibility alias for CCLOAD_API_TOKENS. Do not set it to a different value from CCLOAD_API_TOKENS.
CCLOAD_RELEASE_BASE_URL None Complete .../releases/latest/download base URL used for release detection and downloads. Leave unset to try ghproxy.net first and GitHub directly on failure. Set it only to force a private mirror.
CCLOAD_MYSQL None MySQL DSN. When set, ccLoad uses MySQL; otherwise it uses SQLite. Set it for multiple instances, long-running Hugging Face deployments or external backup.
CCLOAD_ENABLE_SQLITE_REPLICA 0 Hybrid storage switch: write MySQL, read from a SQLite replica. Recommended with MySQL on Hugging Face Spaces.
CCLOAD_SQLITE_LOG_DAYS 7 Number of log days restored from MySQL on hybrid startup. -1 restores all logs; 0 restores no logs.
CCLOAD_ALLOW_INSECURE_TLS 0 Allows skipping upstream TLS certificate verification. Use only for controlled internal debugging. Do not enable in production.
CCLOAD_HOST_OVERRIDES None Pins upstream hostnames to fixed IPs before dialing. Format: host=ip,host2=ip2. Use for upstream DNS failures, for example anyrouter.top=47.246.23.200. TLS SNI, certificates and Host headers stay unchanged. Invalid entries fail startup.
CCLOAD_MAX_CONCURRENCY 1000 Global maximum concurrent proxy requests. Tune it to machine resources and upstream limits. Do not leave concurrency unbounded.
CCLOAD_MAX_BODY_BYTES 10485760 Maximum request body size. Images API automatically raises this to 20MB. Keep the default for public deployments to avoid oversized requests overwhelming the service.

Storage Modes

Choose storage by deployment shape. Do not add MySQL just to look production-grade; SQLite is simpler and has less operational surface for single-instance deployments.

Mode Configuration Use Case Trade-off
SQLite only Do not set CCLOAD_MYSQL Local development, personal deployments and single-instance services. Simplest path; not suitable for shared multi-instance state.
MySQL only Set CCLOAD_MYSQL Standard production, external backups and migration across machines. Strong persistence; read/write latency depends on the database.
Hybrid CCLOAD_MYSQL + CCLOAD_ENABLE_SQLITE_REPLICA=1 Hugging Face Spaces or high-latency MySQL when persistence is still required. Writes go to MySQL and reads use local cache; higher complexity.

Channel Configuration

Channels are ccLoad's core runtime objects. A channel can have multiple keys, multiple upstream URLs, independent cooldown, RPM/concurrency limits and protocol transform policy.

Core fields

  • name: readable channel name used by logs and statistics.
  • type: upstream protocol type such as Anthropic, OpenAI, Gemini or Codex.
  • url: supports comma-separated URLs; a trailing # marks an exact upstream URL and disables automatic path joining.
  • api_key: supports one or multiple keys. ccLoad performs key-level retries inside the channel.
  • priority: higher priority is selected first; equal priority uses smooth weighted round-robin.
  • proxy_url: optional per-channel proxy. Supports http, https, socks5 and socks5h; empty uses the process environment proxy.

Protection fields

  • rpm_limit: rolling 60-second request cap for the channel; 0 means unlimited.
  • max_concurrency: maximum in-flight requests for the channel. Streaming requests release the slot only when the stream ends.
  • daily_cost_limit: daily channel cost limit. The channel is skipped after the limit is reached.
  • cost_multiplier: channel multiplier used for effective-cost limits and accounting.
  • scheduled_check_enabled: enables scheduled probing, usually paired with a check model.
Multiple URLs are channel-internal resilience, not a replacement for channel priority. ccLoad explores URLs first, then selects by 1/EWMA latency weighting. Failed URLs back off independently and do not drag down other URLs in the same channel.
Provider fixed-window quota is treated as a real channel cooldown. When an upstream response includes a concrete reset time, ccLoad waits for that reset instead of guessing with exponential backoff.

Runtime Settings in the Admin Console

These settings are stored in the database and can be adjusted at runtime. They are policy, not startup configuration.

🔁

Retries and cooldown

Key-level errors retry another key inside the channel; 5xx, 598 and 599 channel-level errors trigger channel failover. Cooldown uses exponential backoff and defaults to a 30-minute cap.

⏱️

Protocol timeouts

Supports global first-byte timeout and non-stream timeout, plus Anthropic, Codex, OpenAI and Gemini overrides by upstream protocol.

📈

Health-based sorting

When enabled, ccLoad adjusts effective priority by historical success rate and uses a confidence factor to avoid over-penalizing small samples.

💳

Token limits

Auth tokens can restrict allowed models, allowed channels, cost limits and statistics scope. Costs use integer micro-dollars to avoid floating-point drift.

🧩

Custom request rules

Rewrite headers and JSON bodies per channel with remove, override and append operations. Authentication headers are protected and CRLF injection is blocked.

🧪

Channel checks

Manual and scheduled checks both count toward channel RPM limits. Use a low-cost, stable model for checks.

📚

Model catalog sync

model_catalog_sync_interval_hours refreshes models and pricing from models.dev every 6 hours by default. Set it to 0 to disable network sync; startup falls back from the last-good cache to the embedded catalog.

⬆️

Auto updates

ccLoad checks releases every 12 hours by default, trying ghproxy.net first and GitHub directly on failure. Set auto_update_interval_hours to 0 to disable checks, or use CCLOAD_RELEASE_BASE_URL to force one private mirror.

Security Baseline

The real proxy-service risk is not UI polish; it is exposing upstream keys or the admin surface to the wrong audience.

Pre-production checklist

  • Store the strong password in Secrets, not in the repository.
  • Create at least one API access token; otherwise `/v1/*` returns 401 by design.
  • Restrict admin-console access on public deployments or place it behind a protected network.
  • Do not use `CCLOAD_ALLOW_INSECURE_TLS` as a long-term setting.

Configuration example

.env 
CCLOAD_PASS=your_strong_admin_password
CCLOAD_API_TOKENS=client_token|production

# Optional: MySQL / hybrid mode
CCLOAD_MYSQL=user:pass@tcp(host:3306)/ccload?charset=utf8mb4
CCLOAD_ENABLE_SQLITE_REPLICA=1
CCLOAD_SQLITE_LOG_DAYS=7

# Optional: bypass broken upstream DNS
CCLOAD_HOST_OVERRIDES=anyrouter.top=47.246.23.200
